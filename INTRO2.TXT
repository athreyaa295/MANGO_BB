# **MongoDB & NoSQL Notes**

---

## **1️⃣ NoSQL Databases**

**NoSQL** = “Not Only SQL”

* Designed for **flexible, scalable data storage**.
* Handles **unstructured and semi-structured data** (unlike relational tables).
* Popular for **huge datasets** in big companies.

**Features:**

* Flexible schema (no fixed columns like SQL).
* Fast and efficient storage (uses BSON).
* High scalability (via sharding).
* Handles high volume data.

**Data Types:**

* **Unstructured:** Text, images, videos, etc.
* **Semi-structured:** JSON-like documents.

---

## **2️⃣ MongoDB Overview**

**MongoDB** = A popular **NoSQL database**.

**Key Concepts:**

* Stores data in **documents** (like JSON objects) instead of tables.
* **Documents** are grouped into **collections**.
* Uses **BSON** (Binary JSON) for storage — fast and efficient.
* Supports large data via **sharding** and **replication**.

**Advantages:**

* Can handle huge datasets easily.
* Flexible schema (add/remove fields without altering entire DB).
* Easy to scale horizontally.

**Disadvantages:**

* Requires **more storage space** than relational databases.
* Joins are not native (manual linking or embedding needed).

---

## **3️⃣ RDBMS vs MongoDB (NoSQL)**

| Feature        | RDBMS (MySQL, SQL Server) | MongoDB (NoSQL)         |
| -------------- | ------------------------- | ----------------------- |
| Data Storage   | Tables (rows & columns)   | Documents (JSON/BSON)   |
| Schema         | Fixed                     | Flexible                |
| Relationships  | Foreign Keys, Joins       | Embedding / Manual Ref  |
| Scalability    | Vertical (hardware)       | Horizontal (sharding)   |
| Query Language | SQL                       | MongoDB Query Language  |
| Suitable for   | Structured data           | Unstructured / Big data |

---

## **4️⃣ Keys in Databases**

### **Primary Key**

* Unique identifier for a record in a table.

### **Foreign Key**

* Column in one table referring to the **primary key** of another table.
* Used to maintain **relationships between tables** in SQL.

> **Note:** MongoDB doesn’t enforce foreign keys; relationships are handled via **embedding** or **manual references**.

---

## **5️⃣ MongoDB Data Types**

* **BSON:** Binary JSON, the format MongoDB uses to store data efficiently.
* **JSON:** JavaScript Object Notation, human-readable format similar to BSON.

**Example Document in MongoDB:**

```js
{
  name: "React",
  videos: 35,
  duration: 11,
  completed: true,
  PublishedDate: new Date(),
  Instructor: ["Goms", "Siva"],
  relatedCourses: { name: "JS", videos: 36, duration: 11 }
}
```

---

## **6️⃣ MongoDB Structure**

* **Database:** Collection of collections.
* **Collection:** Group of documents (like a table).
* **Document:** A single record (like a row), stored in **BSON**.

**Example:**

```text
Database: codeio
Collection: courses
Document: {name: "React", videos: 35}
```

## **7️⃣ MongoDB Commands**

### **Switching Database**

```js
use codeio
```

### **Show Databases**

```js
show dbs
```

### **Create Collection**

```js
db.createCollection("courses")
```

### **Show Collections**

```js
show collections
```

### **Insert Document**

```js
db.courses.insertOne({
  name: "React",
  videos: 35,
  duration: 11
})
```

### **Find Document**

```js
db.courses.find()
db.courses.find().pretty()
```

## **8️⃣ MongoDB Features**

### **Sharding**

* Splitting a huge dataset into smaller parts called **shards**.
* Improves **query performance**.
* Example: 10,000 records split into multiple shards for faster access.

### **Replication**

* Creates **duplicates of data** across servers.
* Ensures **data availability and fault tolerance**.

  * Monitor performance

## **10️⃣ Summary Notes**

* **MongoDB = NoSQL database**
* **Data stored in documents, not tables**
* **BSON** = Binary JSON for efficient storage
* **Sharding** → performance
* **Replication** → reliability
* **No enforced foreign keys**, use embedding or references instead
* **RDBMS vs NoSQL:** SQL uses tables & columns; NoSQL uses flexible documents
